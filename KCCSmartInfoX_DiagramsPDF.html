Figure 1
Conceptual Framework — IPO Model
Shows the overall Input–Process–Output model of KCCSmartInfoX. Inputs come from both the Student (questions, feedback, concerns) and the Administrator (documents, announcements, credentials). The Process layer includes authentication, the RAG AI pipeline, document ingestion, email delivery, and analytics. Outputs include AI-generated answers with source citations, announcement emails, chat logs, and audit reports.

Figure 2
System Architecture Diagram
Shows the three-tier architecture of KCCSmartInfoX. The Client Layer (React 18 + Vite + TailwindCSS + PWA) communicates via HTTP REST API to the Backend Layer (FastAPI + JWT Auth + Rate Limiter + RAG Pipeline + Background Tasks), which connects to the Data Layer (SQLite, ChromaDB, Groq API, Gmail SMTP).

Figure 3
Use Case Diagram
Shows the two actors — Student and Administrator — and all use cases available to each. Students can ask questions, submit concerns, give feedback, subscribe, use voice input, switch languages, and install as PWA. Administrators can manage the knowledge base, post announcements, view analytics, generate FAQ suggestions, reply to concerns, view chat logs, manage office processes, generate QR codes, and manage account security.

Figure 4
Data Flow Diagram — Level 0 (Context Diagram)
The highest-level view of the system. KCCSmartInfoX is shown as a single process with four external entities: Student (sends questions and feedback; receives AI answers), Administrator (sends documents and settings; receives analytics), Groq API (receives AI prompts; returns generated answers), and Gmail SMTP (receives email content; delivers to subscribers and students).

Figure 5
Data Flow Diagram — Level 1
Breaks the system into six main processes: (1) Authentication — JWT login and audit logging; (2) Chat/RAG Processing — AI pipeline with ChromaDB and Groq; (3) Knowledge Base Management — document ingestion into SQLite and ChromaDB; (4) Announcement Management — scheduled posts and email delivery; (5) Analytics and Reporting — statistics and CSV exports; (6) Concern Management — student submissions and admin email replies.

Figure 6
RAG Pipeline Flowchart — AI Query Processing
The step-by-step AI query processing flow: thanks detection → greeting detection → spell correction (pyspellchecker) → LLM query rewriting (Groq, temp=0.0) → semantic embedding (all-MiniLM-L6-v2, 384-dim) → ChromaDB vector search (top 3, cosine similarity) → distance check (≤1.3) → context trimming (150 words) → Groq LLM answer generation (llama-3.1-8b-instant, temp=0.2, max=400) → bullet cleanup → is_answered check → SQLite logging → return to student.

Figure 7
Entity Relationship Diagram (ERD)
The database structure of KCCSmartInfoX with 10 tables: admin_users (bcrypt-hashed credentials), chat_logs (all Q&A records), feedback (thumbs up/down linked to chat_logs), subscribers (email list), announcements (with scheduling), knowledge_docs (uploaded files), office_processes (JSON sections), concerns (student submissions with admin replies), admin_login_logs (audit trail), and admin_ai_logs (AI assistant history). Key relationships: feedback → chat_logs (FK), concerns ↔ chat_logs (related question).
