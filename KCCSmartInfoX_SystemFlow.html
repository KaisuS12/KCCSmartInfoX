Chapter 1  ·  System Overview
1.1   Description of the System

KCCSmartInfoX is a web-based artificial intelligence information assistant developed for Kabankalan Catholic College, Inc. (KCC). The system is designed to provide students with immediate, accurate, and accessible answers to their school-related questions at any time of the day, without requiring them to visit a school office or wait for a staff member to respond. The system uses Retrieval-Augmented Generation (RAG), a modern AI technique that grounds the chatbot's answers strictly in official KCC documents rather than allowing the AI to generate unreliable or fabricated information.

The system is composed of two main interfaces: the Student Interface, which is a chat-based platform accessible through any web browser or as an installable mobile application, and the Admin Panel, which is a secured management dashboard where school administrators can manage the knowledge base, post announcements, monitor system activity, and respond to student concerns. Both interfaces are connected to a shared backend server and database, ensuring that all information displayed to students is up to date and consistent with what the administration has provided.

1.2   System Architecture

The architecture of KCCSmartInfoX follows a three-tier client-server model. The first tier is the frontend, built using React 18 and Vite, which handles all user interactions and renders the chat interface and admin panel in the browser. The second tier is the backend, built using FastAPI (a Python web framework), which processes all requests, manages authentication, communicates with the database, and coordinates the AI pipeline. The third tier consists of the data layer, which includes a SQLite relational database for storing structured data such as user accounts, chat logs, announcements, and concerns, and a ChromaDB vector database for storing document embeddings used in the AI retrieval process.

Communication between the frontend and backend is done through a RESTful API, where the frontend sends HTTP requests and the backend responds with JSON data. All admin routes are protected using JSON Web Token (JWT) authentication, ensuring that only authorized administrators can access sensitive operations. The AI capabilities are powered by the Groq API, which hosts the LLaMA 3.1-8B Instant language model for fast and cost-efficient text generation.

Chapter 2  ·  User and Student Flow
2.1   Accessing the System

A student accesses KCCSmartInfoX by opening the system's URL in any modern web browser on a computer or mobile phone. Upon loading, the student is directed to the Landing Page, which presents an announcement carousel showing the latest school announcements and a section displaying the college's office processes. This page provides students with an immediate overview of current school news before they even open the chatbot. From the landing page, the student can click the provided navigation button to open the chat interface.

For students using mobile devices, KCCSmartInfoX is configured as a Progressive Web App (PWA), meaning it can be installed directly on the home screen of a smartphone like a native application. This eliminates the need to open a browser and type the URL each time, making the system more accessible and convenient for daily student use.

2.2   Using the Chat Interface

Once the chat interface is opened, the student is greeted by the KCCSmartInfoX chatbot. The interface displays a set of suggested starter questions to help students who are unsure what to ask, covering common concerns such as enrollment procedures, tuition fees, and transcript requests. The student may click any of these suggestions or type their own question in the input field.

The system supports two language modes: English and Filipino. The student can toggle between these languages using the language switch button in the chat header. When Filipino mode is selected, the AI assistant responds in Filipino or Taglish, making the system more inclusive and accessible to students who are more comfortable communicating in their native language. The selected language preference is saved locally so it persists across sessions.

For students who prefer speaking over typing, the system includes a voice input feature accessible through a microphone button in the chat interface. When activated, the system uses the browser's built-in Speech Recognition API to convert the student's spoken words into text, which is then submitted as a question. This feature is currently supported on Google Chrome.

The student may also open two side panels from the chat header: the Announcements Panel, which displays all currently active school announcements, and the School Information Panel, which displays general KCC contact details and office hours. Both panels slide in without leaving the chat page, allowing the student to reference information without losing their conversation.

2.3   Submitting a Question and Receiving an Answer

When a student submits a question, it is sent to the backend server for processing through the AI pipeline (described in detail in Chapter 4). While the AI processes the question, a typing indicator is displayed in the chat so the student knows the system is working. Once the answer is generated, it appears in the chat window in a readable format, with bullet points and headings properly rendered for clarity.

If the AI was able to answer the question using KCC official documents, the source documents used to generate the answer are displayed below the response. This allows the student to verify the source of the information and builds trust in the accuracy of the system's answers. The system also suggests follow-up questions based on the topic of the answer, encouraging the student to explore related information without having to think of what to ask next.

If the AI is unable to answer the question because no relevant information was found in the knowledge base, it responds honestly by informing the student that it does not yet have information on that topic and that the question has been noted for review. In this case, a "Submit a Concern" button appears below the response, inviting the student to escalate their query directly to the admin.

2.4   Providing Feedback

Below each AI-generated answer, the student is provided with thumbs-up and thumbs-down feedback buttons. If the student found the answer helpful, they may click the thumbs-up button; if the answer was poor or inaccurate, they may click the thumbs-down button. This feedback is recorded in the database and is visible to the administrator in the Analytics page, where it helps identify which answers need improvement and which topics are being handled well by the system.

2.5   Submitting a Concern

When the AI cannot answer a student's question, the student is given the option to submit a formal concern directly through the system. Clicking the "Submit a Concern" button opens a modal form within the chat interface. The form automatically pre-fills the student's unanswered question in the message field. The student then provides their name and email address and may edit or expand their message before submitting.

Upon submission, the concern is saved to the database with a status of "Pending" and an automated email notification is sent to the administrator, informing them that a new concern has been received and providing the full details. The student is shown a confirmation message that their concern has been submitted and that the administrator will reply to their email address. This closes the loop between the student's unanswered question and the admin's ability to respond personally.

2.6   Subscribing to Announcements

Students may subscribe to the KCCSmartInfoX announcement system by entering their email address in the subscription field available in the chat interface. Once subscribed, the student's email is recorded in the database. Whenever the administrator posts a new announcement, all subscribers automatically receive the announcement via email, ensuring that students stay informed about important school news even when they are not actively using the system.

Chapter 3  ·  Administrator Flow
3.1   Admin Login and Session Management

The administrator accesses the admin panel by navigating to the designated admin login URL. The login page requires a username and password, which are verified against the stored credentials in the database. Passwords are stored using bcrypt hashing, a security standard that ensures plain-text passwords are never stored or exposed. The login endpoint is rate-limited to five requests per minute per IP address to prevent brute-force attacks. Upon successful authentication, the server generates a JSON Web Token (JWT) with a 24-hour expiration period. This token is stored in the browser's local storage and is automatically included in every subsequent admin API request as an authorization header.

Every login attempt — whether successful or failed — is recorded in the admin audit log with the event type, the administrator's username, the IP address of the request, the device type derived from the user agent, and a timestamp. When the administrator manually logs out, a logout event is also recorded. If the admin is automatically signed out due to inactivity (described below), a timeout event is recorded instead. This audit log is visible to the administrator in the Settings page, providing full visibility into all account activity.

If the token expires or is invalidated while the admin is using the panel, any request to a protected endpoint returns an HTTP 401 Unauthorized response. The system detects this through a global Axios response interceptor and immediately redirects the administrator to the login page, clearing the expired token. Additionally, the admin panel tracks user activity events — mouse movement, keystrokes, clicks, and scrolling. If no activity is detected for thirty consecutive minutes, the session is automatically terminated, a timeout event is logged, and the administrator is redirected to the login page. The inactivity timer resets on each detected activity event.

3.2   Dashboard

After logging in, the administrator is taken to the Dashboard, which serves as the central overview of the system's activity. The Dashboard displays key metrics including the total number of questions asked, the number of answered and unanswered questions, the overall answer rate expressed as a percentage, the total number of email subscribers, the total number of uploaded knowledge documents, the number of positive feedback votes, and the number of pending student concerns. These statistics are fetched from the backend automatically every ten seconds, ensuring the admin always sees the most current data without manually refreshing the page.

The Dashboard also displays a list of recent unanswered questions, giving the administrator a quick view of what students are asking that the AI cannot yet answer. Quick-action buttons allow the administrator to navigate directly to the most commonly used pages: adding knowledge, viewing analytics, posting an announcement, or reviewing concerns. A real-time Backend Online/Offline status indicator is also displayed in the Dashboard header, showing whether the FastAPI server is reachable. This is particularly useful during local network demonstrations where multiple devices share the same backend.

3.3   Managing the Knowledge Base

The Knowledge Base page is the core of the system's AI capability. It is where the administrator uploads the documents that the AI uses to answer student questions. The administrator may upload individual files in PDF, DOCX, or TXT format, or upload multiple files at once using the bulk upload feature. After uploading, the system automatically extracts the text content from the file, splits it into smaller chunks, converts each chunk into a numerical vector using a sentence embedding model, and stores these vectors in the ChromaDB vector database. This process is called indexing, and once complete, the document's content is immediately searchable by the AI.

In addition to file uploads, the administrator may also add text entries directly without a file. This is useful for adding short pieces of information such as updated schedules, policies, or contact details that do not exist as a formal document. These text entries are stored in the relational database and indexed into ChromaDB in the same way as uploaded files. Text entries can also be edited inline within the Knowledge Base page — when edited, the old indexed content is removed and replaced with the new content automatically. Documents and text entries can also be deleted, which simultaneously removes the record from the database and the corresponding chunks from the vector store.

3.4   Managing Announcements

The Announcements page allows the administrator to create, edit, and delete school announcements. When creating an announcement, the administrator provides a title and content body, and may optionally attach an image. The administrator may choose to publish the announcement immediately or set a future date and time for it to go live automatically. An expiry date may also be set, after which the announcement is automatically hidden from the student-facing landing page and chat panel without any manual action required.

When an announcement is published — whether immediately or upon reaching its scheduled time — the system automatically sends the announcement to all subscribed students via email. The email is sent in the background as a separate task so that the administrator does not experience any delay in the response when posting. For scheduled announcements, a background process running every sixty seconds checks whether any pending announcements have reached their publish time and triggers the email delivery accordingly.

3.5   Viewing Analytics

The Analytics page provides the administrator with a detailed view of how students are using the system. It displays a bar chart showing the number of questions asked per day over the last seven days, a ranked list of the ten most frequently asked questions, a complete list of all unanswered questions, and a list of answers that students rated negatively using the thumbs-down button. The administrator can export the list of unanswered questions as a CSV file for use in reports or meetings.

The Analytics page also includes the FAQ Auto-Generator feature. When the administrator clicks the Generate button, the system sends the most recent unanswered questions to the Groq AI, which analyzes them and suggests knowledge base entries that would help answer those questions. The suggestions are displayed as draft content that the administrator can review, edit, and add directly to the knowledge base with a single click. This feature significantly reduces the time needed to identify and fill knowledge gaps in the system.

3.6   Managing Student Concerns

The Concerns page displays all concerns submitted by students through the chat interface. Each concern shows the student's name, email address, the message they submitted, the related unanswered question (if any), the date and time of submission, and a status badge indicating whether the concern is Pending or Resolved. The administrator may filter concerns by status to focus on those that still require attention.

To respond to a concern, the administrator clicks on the concern card to expand it, reads the full message, and types a reply in the provided text area. Upon clicking "Send Reply," the reply is saved to the database, the concern's status is updated to Resolved, and the reply is automatically sent to the student's email address as a background task. If the administrator wishes to mark a concern as resolved without sending a reply — for example, if the issue was handled through a different channel — they may use the "Mark Resolved" button instead. Concerns may also be deleted from the system entirely.

3.7   Other Administrative Functions

The School Info page allows the administrator to edit the school information sections that appear in the student-facing School Information side panel in the chat interface. Any edits made here are immediately reflected on the student side without requiring a page refresh or system restart.

The Office Processes page allows the administrator to add, edit, and delete the office processes displayed on the landing page. Crucially, every change made to office processes is automatically re-synchronized to the ChromaDB vector database, ensuring that the AI can answer questions about office procedures based on the most current information without requiring a manual re-upload of documents.

The Chat Logs page gives the administrator full visibility into every question asked through the chat interface, up to the most recent two hundred entries. The three summary cards at the top — Total Questions, Answered, and Unanswered — act as clickable filter buttons that instantly filter the list below, removing the need for a separate filter control. Logs can also be searched by keyword. Each log entry can be expanded to view the AI's full response. The unanswered logs can be exported as a CSV file. The Subscribers page lists all email subscribers with their registration dates and allows the administrator to remove individual subscribers. The QR Code Generator page allows the administrator to create a QR code for the chat URL, automatically detecting the local network IP address and the active port, which the admin can print or display for students to scan. The Settings page allows the administrator to change their account password (requiring the current password for verification) and view the Admin Audit Log, which shows the last fifty login, logout, and session timeout events with device and IP details.

Chapter 4  ·  AI Query Processing Flow
4.1   Overview of Retrieval-Augmented Generation (RAG)

KCCSmartInfoX uses a technique called Retrieval-Augmented Generation, commonly referred to as RAG, to power its AI chatbot. Unlike a standard language model that generates answers purely from its training data — which may be outdated or hallucinated — a RAG system first retrieves relevant factual content from a trusted knowledge source, then uses that content as context when generating a response. In the case of KCCSmartInfoX, the trusted knowledge source is the collection of official KCC documents uploaded by the administrator. This design ensures that the AI's answers are grounded in actual KCC information and cannot fabricate details such as tuition fees, deadlines, or office names that do not exist in the documents.

4.2   Step-by-Step Query Processing
Step 1: Receiving the Raw Question

The process begins when a student submits a question through the chat interface. The question is sent to the backend server along with the student's selected language (English or Filipino) and the last four messages in the conversation, which are used to maintain context across multi-turn conversations. The question may contain typos, Taglish expressions, abbreviations, or incomplete sentences, all of which the system is designed to handle gracefully.

Step 2: Thank-You and Greeting Detection

Before performing any AI processing, the system first checks whether the student's input expresses gratitude — messages such as "thank you," "thanks," "salamat," "maraming salamat," or "daghan salamat." This check is done using a set of predefined text patterns and requires no API call. If the message is a thank-you, the system responds immediately with one of several warm, randomized reply messages in the student's chosen language and bypasses the rest of the pipeline entirely. This detection runs before the greeting check to ensure gratitude messages are handled naturally rather than being treated as general small talk.

If the message is not a thank-you, the system then checks whether it is a casual greeting — such as "hello," "hi," "kumusta," or "good morning." If a greeting is identified, the system sends it directly to the language model with a simple conversational prompt, which returns a warm and welcoming response. Both checks improve system efficiency by avoiding unnecessary document searches for messages that are not genuine information queries.

Step 3: Spell Correction

If the message is an information query, the system first applies a local spell checker to correct common English typographical errors. The spell checker is configured with a custom word list of KCC-specific and Filipino terms — such as "enrollment," "registrar," "promissory," "bsit," "tor," "kumusta," and "po" — so that these terms are never incorrectly modified. This step runs locally on the server without any API call, making it fast and cost-free.

Step 4: LLM Query Rewriting

After spell correction, the cleaned question is sent to the Groq language model with a specialized rewriting prompt. The purpose of this step is to convert the student's raw input — which may still contain Taglish phrases, Bisaya words, abbreviations, or grammatical errors — into a clean, precise English question that will produce better results during the vector search. For example, the input "pano kumuha ng tor sa rejistrar" is rewritten as "How do I get a Transcript of Records from the registrar?" This rewriting step significantly improves the accuracy of the subsequent document retrieval, as the semantic search works best with well-formed English queries.

Step 5: Semantic Embedding

The rewritten question is then converted into a numerical vector representation using the all-MiniLM-L6-v2 sentence-transformers model, which runs locally on the server. This model encodes the semantic meaning of the question into a fixed-length vector of numbers. Unlike keyword-based search, this vector captures the meaning of the question rather than just its words, which means it can find relevant documents even when the student uses different wording than what appears in the documents.

Step 6: ChromaDB Vector Search

The question vector is compared against all document chunk vectors stored in the ChromaDB vector database using cosine similarity, a mathematical measure of how similar two vectors are in meaning. The system retrieves the three most similar document chunks along with their similarity scores. If the closest matching document chunk has a distance score greater than 1.3 — indicating that even the best match is too dissimilar to be relevant — the system determines that the question cannot be answered from the current knowledge base and flags it as unanswered. Otherwise, the retrieved chunks are passed to the next step.

Step 7: Answer Generation

The retrieved document chunks, together with the original question, the rewritten question, and the last four messages of the conversation history, are assembled into a prompt and sent to the Groq LLaMA 3.1-8B Instant language model. The system prompt instructs the model to answer strictly based on the provided KCC document context, to use proper markdown formatting for readability, never to use bullet characters (only dash-space for bullet lists), to keep answers concise and mobile-friendly, and to respond in the student's chosen language (English or Filipino). The model generates an answer within the constraints of the provided context and the formatting rules. The entire Groq API call is wrapped in error handling — if the API is temporarily unavailable or returns an error, the system returns a user-friendly fallback message informing the student to try again shortly, rather than displaying a raw error.

Step 8: Post-Processing and Storage

The generated answer undergoes a post-processing step that normalizes any bullet characters into proper markdown format, ensuring consistent rendering in the frontend. The system then checks whether the answer contains the phrase "I don't have information about that yet," which is the model's instructed response when it cannot find an answer in the context. Based on this check, the answer is labeled as either answered or unanswered. The question, answer, and answered status are saved as a new record in the chat_logs table of the SQLite database. The final response — including the answer text, the answered flag, the source document names, and the chat log ID — is returned to the frontend, which renders it in the chat window.

Chapter 5  ·  Data Management and Storage
5.1   Relational Database (SQLite)

The system uses a SQLite relational database to store all structured data. The database contains ten tables: admin_users, which stores the administrator account credentials with bcrypt-hashed passwords; chat_logs, which records every question and answer exchanged through the chat interface along with the answered status and timestamp; feedback, which stores the thumbs-up and thumbs-down ratings given by students to individual answers; subscribers, which stores the email addresses of students who have subscribed to announcements; announcements, which stores all announcement content including title, body, publish and expiry dates, image filenames, and email delivery status; knowledge_docs, which stores metadata for all uploaded documents and manual text entries; school_info, which stores the editable sections of school information displayed in the chat sidebar; office_processes, which stores the office process records managed through the admin panel; concerns, which stores all student concerns including the submitted message, admin reply, and resolution status; and admin_login_logs, which records every admin login attempt, logout, and session timeout event with the username, event type, IP address, device type, and timestamp.

5.2   Vector Database (ChromaDB)

In addition to the relational database, the system uses ChromaDB, a local vector database, to store the semantic embeddings of all knowledge base content. ChromaDB stores each document chunk as a vector along with the original text and the source filename as metadata. When a student asks a question, the system queries ChromaDB to find the most semantically similar chunks, which are then used as context for the AI answer generation. The ChromaDB files are stored locally on the server in a persistent directory, ensuring that indexed content survives server restarts without needing to be re-indexed. A shared singleton client instance is used throughout the application to prevent database lock conflicts that would occur if multiple parts of the application attempted to open the database simultaneously.

5.3   File Storage

Uploaded knowledge base documents are stored in the local file system under the knowledge_base/documents directory. Announcement images are stored under the knowledge_base/announcement_images directory. PWA icons used for the installable mobile application are stored in the frontend's public/icons directory. These file storage paths are consistent across the application and are referenced by the database records that correspond to each file.

Chapter 6  ·  Technology Stack
6.1   Frontend Technologies

The frontend of KCCSmartInfoX is developed using React 18, a JavaScript library for building interactive user interfaces, with Vite 5 as the build tool for fast development and optimized production builds. Styling is handled by TailwindCSS 3, a utility-first CSS framework that enables rapid and consistent design. React Router DOM manages client-side navigation between pages. Axios is used for making HTTP requests to the backend API. ReactMarkdown renders the AI's formatted responses with proper headings, bullet lists, and code blocks. Recharts provides the bar and line charts displayed on the Analytics page. Lucide React supplies the icon set used throughout the interface. React Hot Toast displays non-intrusive notification messages for user actions. The vite-plugin-pwa package configures the Progressive Web App functionality, enabling the system to be installed on mobile devices.

6.2   Backend Technologies

The backend is built with Python 3.11 and FastAPI, an asynchronous web framework that provides automatic API documentation and high performance. SQLAlchemy is used as the Object-Relational Mapper (ORM) for database operations, with SQLite as the underlying relational database engine. ChromaDB serves as the vector database for storing and querying document embeddings. The sentence-transformers library, specifically the all-MiniLM-L6-v2 model, generates the semantic embeddings used for document retrieval. The pdfplumber library handles PDF text extraction, and python-docx handles DOCX file parsing. Authentication is implemented using python-jose for JWT token creation and verification, and passlib with bcrypt for secure password hashing. The slowapi library enforces rate limiting on the chat endpoint to prevent abuse. Email delivery is handled using Python's built-in smtplib library with Gmail's SMTP SSL service.

6.3   AI and External Services

The Groq API serves as the host for the LLaMA 3.1-8B Instant language model, which is used for both query rewriting and answer generation. Groq was selected for its high-speed inference capabilities, which allow the system to generate responses quickly without significant latency. The all-MiniLM-L6-v2 model from the sentence-transformers library runs locally on the server and does not require an external API call, reducing dependency on external services for the retrieval component of the RAG pipeline. The pyspellchecker library provides local spell correction without an API call. The Web Speech API, built into Google Chrome, powers the voice input feature on the student side.
