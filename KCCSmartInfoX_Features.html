Part 1  ·  Student-Facing Features

The student-facing side of KCCSmartInfoX is designed for accessibility, speed, and ease of use on any device. The following features are available to all students without requiring an account or login.

Feature 1  —  AI-Powered Question Answering

The core feature of KCCSmartInfoX is its artificial intelligence chatbot that answers student questions about Kabankalan Catholic College. Students type their question in the chat interface and receive a generated response within seconds. The system uses Retrieval-Augmented Generation (RAG) — a technique that first searches official KCC documents for relevant information before generating an answer, ensuring that all responses are grounded in actual school policies and records rather than fabricated content. If the system is unable to find a relevant answer in the knowledge base, it honestly informs the student and flags the question for administrator review.

Feature 2  —  Suggested Starter Questions

Upon opening the chat interface, students are presented with four suggested starter questions displayed as clickable cards below the KCC logo. These suggestions cover the most commonly asked topics — enrollment procedures, Transcript of Records, Discipline Office policies, and office hours — and allow students to begin asking questions with a single tap, eliminating hesitation for first-time users. Clicking any suggestion card immediately submits the question to the AI.

Feature 3  —  Follow-Up Question Suggestions

After the AI generates a response, the system analyzes the topic of the answer and displays up to three related follow-up questions below the response. These suggestions are dynamically generated based on keywords in the AI's answer, such as enrollment, tuition, scholarship, or transcript. This feature encourages students to continue exploring related topics without having to think of what to ask next, improving the depth of information they receive in a single session.

Feature 4  —  English and Filipino Language Toggle

KCCSmartInfoX supports two language modes: English and Filipino. Students can switch between these modes using the language toggle button in the chat header. When Filipino mode is selected, the AI responds in Filipino or Taglish, making the system more inclusive and accessible to students who are more comfortable communicating in their native language. The system also understands Bisaya and Taglish input regardless of which mode is active. The selected language preference is saved in the browser's local storage so it persists across sessions without requiring the student to switch again.

Feature 5  —  Voice Input

Students who prefer speaking over typing may use the microphone button in the chat input area to submit questions through voice. When activated, the system uses the browser's built-in Web Speech API to transcribe the student's spoken words into text in real time. The transcribed text is then placed in the input field, which the student can review and edit before submitting. This feature is currently supported on Google Chrome and makes the system more accessible to students with difficulty typing.

Feature 6  —  Thank-You Detection and Instant Reply

The system recognizes when a student expresses gratitude — through messages such as "thank you," "thanks," "salamat," "maraming salamat," "daghan salamat," "ty," or similar expressions — and responds immediately with a warm, randomized reply without performing any document search. Five different reply templates are available in both English and Filipino, ensuring that the response feels natural and varied rather than repetitive. This detection runs before any other processing step, making it the fastest type of response in the system.

Feature 7  —  Greeting Recognition

When a student opens a conversation with a casual greeting — such as "hello," "hi," "kumusta," "good morning," "magandang umaga," or similar expressions — the system detects this through pattern matching and responds with a warm, conversational welcome message without performing a document search. The greeting response is generated by the Groq language model with a conversational prompt, making it feel friendly and natural. This detection prevents the system from unnecessarily searching the knowledge base for messages that are not genuine information queries.

Feature 8  —  Source Citation

When the AI generates an answer using content from uploaded KCC documents, the filenames of the source documents are displayed below the response. This allows students to see exactly which official document was used to generate the answer — for example, "KCC College Handbook 2025-2026 Final.pdf" — building trust in the accuracy of the information and allowing students to verify the source if needed. No sources are shown for greeting and thank-you responses since no documents are used in those cases.

Feature 9  —  Thumbs Up and Thumbs Down Feedback

Below each AI-generated answer, students are provided with thumbs-up and thumbs-down feedback buttons. Clicking thumbs-up indicates that the answer was helpful, while thumbs-down indicates that it was not. This feedback is recorded in the database and displayed in the administrator's Analytics page, where it helps identify which answers need improvement and which topics the system is handling well. Each feedback entry is linked to the specific chat log record it rates.

Feature 10  —  Retry Button on Error

If the AI returns an error — due to a temporary network issue or Groq API unavailability — the error message displayed in the chat includes a Retry button. Clicking this button automatically resubmits the student's original question without requiring them to retype it. This ensures a smooth experience even when brief connectivity issues occur.

Feature 11  —  Character Counter and 500-Character Limit

The chat input field enforces a maximum length of 500 characters per message. A live character counter displayed below the input field shows the current character count in relation to the limit. The counter text turns yellow when the student approaches 350 characters and turns red when they reach 450 characters, providing a clear visual warning before the limit is reached. This limit prevents excessively long inputs that could affect AI response quality.

Feature 12  —  Copy Answer to Clipboard

Each AI-generated answer includes a copy button that allows the student to copy the full answer text to their clipboard with a single click. A visual checkmark confirmation appears briefly after copying. This feature is useful for students who wish to save an answer for reference or share it with a classmate.

Feature 13  —  Submit a Concern

When the AI cannot answer a student's question, a contact card appears below the response with a "Submit a Concern" button. Clicking this button opens a modal form within the chat interface. The form automatically pre-fills the student's unanswered question in the message field. The student provides their name and email address and may expand or edit the message before submitting. Upon submission, the concern is saved to the database with a status of "Pending" and an automated email notification is sent to the administrator. The student receives a confirmation message that their concern has been submitted and that a reply will be sent to their email address.

Feature 14  —  Announcements Side Panel

Students can open an Announcements panel from the chat header by clicking the Announcements button. The panel slides in and displays all currently active school announcements fetched from the database, showing each announcement's title, content, image (if any), and publication date. Students can browse announcements without leaving the chat interface. The panel can be closed with an X button, returning the student to their conversation.

Feature 15  —  Subscribe to Announcements

Students may subscribe to the KCCSmartInfoX announcement system by entering their email address in the subscription field available in the chat interface. Once subscribed, their email address is stored in the database. Whenever the administrator posts a new announcement, all subscribed students automatically receive the full announcement content via email, ensuring they stay informed about important school news even when they are not actively using the system.

Feature 16  —  Dark Mode and Light Mode

The chat interface supports both dark and light color themes. Dark mode is enabled by default, featuring a deep navy background that reduces eye strain during extended nighttime use. Students can toggle to light mode using the sun/moon icon button in the chat header. The selected theme preference is saved in the browser's local storage so it is remembered between sessions.

Feature 17  —  Landing Page with Announcement Carousel

The system's landing page serves as the entry point for students before they open the chatbot. It features a navigation bar with the KCC logo, a hero section with a prominent call-to-action button to open the chatbot, an announcement carousel displaying the latest active school announcements fetched from the database, and a grid of office process cards. The landing page provides students with an immediate overview of current school news and common office procedures upon arrival.

Feature 18  —  Office Process Cards on Landing Page

The landing page displays a grid of office process cards representing the different school offices — such as the Registrar, Cashier, Guidance Office, and Library. Each card is managed by the administrator through the admin panel and shows the office name and a short description of its processes. Clicking a card on the landing page can guide the student to ask the chatbot about that specific office. All office process data is fetched dynamically from the backend and is always current.

Feature 19  —  Progressive Web App (PWA) — Installable on Mobile

KCCSmartInfoX is configured as a Progressive Web App (PWA), which means students can install it directly on the home screen of their smartphone or tablet without going through an app store. On Android devices, the browser prompts users to add the app to their home screen; on iOS devices, this can be done through the Safari Share menu. Once installed, the app launches in full-screen mode with the KCC logo as the app icon, providing a native app-like experience. This makes the system significantly more accessible for daily student use.

Part 2  ·  Administrator Features

The administrator panel is a secured management dashboard accessible only through JWT-authenticated login. It provides the school administrator with complete control over the system's knowledge base, announcements, and monitoring tools.

Feature 20  —  Secure Admin Login with Rate Limiting

The administrator accesses the admin panel through a dedicated login page that requires a username and password. Passwords are stored in the database using bcrypt hashing — a one-way cryptographic algorithm that ensures the original password is never stored or exposed. Upon successful login, the server generates a JSON Web Token (JWT) with a 24-hour expiration period that is automatically included in all subsequent admin requests. The login endpoint is rate-limited to five attempts per minute per IP address using the slowapi library, preventing brute-force password attacks. Both successful and failed login attempts are recorded in the audit log.

Feature 21  —  Auto-Logout on Inactivity

The admin panel automatically signs out the administrator after thirty consecutive minutes of inactivity. The system tracks five types of user activity — mouse movement, keystrokes, mouse clicks, touch events, and scrolling. Each detected activity resets the inactivity timer. When the timer expires without any activity, the session is terminated, a "Session Timeout" event is recorded in the audit log with a timestamp and device details, and the administrator is redirected to the login page. This feature protects the admin panel from unauthorized access in cases where the administrator forgets to log out.

Feature 22  —  Admin Dashboard with Real-Time Statistics

The Dashboard is the first page the administrator sees after logging in. It displays eight key metrics: total number of questions asked, number of answered questions, number of unanswered questions, overall answer rate expressed as a percentage, total number of email subscribers, total number of uploaded knowledge documents, number of positive feedback votes, and number of pending student concerns. All statistics are fetched from the backend every ten seconds automatically, ensuring the administrator always sees the most current data. The Dashboard also displays a Backend Online/Offline status indicator and a list of recent unanswered questions for quick reference.

Feature 23  —  Knowledge Base Management

The Knowledge Base page is where the administrator manages all the documents and text entries that the AI uses to answer student questions. The administrator may upload individual files in PDF, DOCX, or TXT format, or upload multiple files at once using the bulk upload feature. Each file is subject to a 20-megabyte size limit. After uploading, the system automatically extracts the text content, splits it into chunks of approximately 400 words with 50-word overlaps, generates 384-dimensional semantic embeddings for each chunk, and stores them in the ChromaDB vector database — a process called indexing. The administrator may also add text entries directly without uploading a file, which is useful for short policy updates or schedule changes. Text entries can be edited inline, and both file uploads and text entries can be deleted, which simultaneously removes the record from the database and the corresponding embeddings from ChromaDB.

Feature 24  —  Announcement Management with Scheduling

The Announcements page allows the administrator to create, edit, and delete school announcements. When creating an announcement, the administrator provides a title and content body, and may optionally attach an image. An optional publish date and time can be set for future announcements, and an expiry date can be set after which the announcement is automatically hidden from the student-facing pages without any manual action. When an announcement is published, the system automatically sends the full announcement to all subscribed students via email as a background task, ensuring the API responds immediately without waiting for the email delivery to complete. Existing announcements can be edited through a pencil icon that opens a pre-filled edit modal.

Feature 25  —  Analytics Dashboard

The Analytics page provides the administrator with a detailed view of system usage. It displays a bar chart showing the number of questions asked per day over the last seven days, a ranked list of the ten most frequently asked questions, a complete list of all unanswered questions, and a list of answers that students rated as not helpful using the thumbs-down button. All of these views are organized under tabs that fade in smoothly when switched. The administrator can export the chat logs — filtered by all, answered, or unanswered — as a CSV file containing the question, answer, status, and timestamp of each entry.

Feature 26  —  FAQ Auto-Generator

The Analytics page includes an FAQ Auto-Generator feature that uses artificial intelligence to help the administrator fill knowledge gaps in the system. When the administrator clicks the Generate button, the system sends the most recent unanswered student questions to the Groq language model, which analyzes them and suggests knowledge base entries that would answer those questions. The suggested content is displayed as draft text that the administrator can review, edit, and add directly to the knowledge base with a single click. This feature significantly reduces the time and effort required to identify and address gaps in the AI's knowledge.

Feature 27  —  Student Concerns Management

The Concerns page displays all concerns submitted by students through the chat interface. The three summary cards at the top of the page — showing total, pending, and resolved concern counts — function as clickable filter buttons that instantly filter the list below. Each concern card shows the student's name, email, message, related unanswered question, submission date, and current status. The administrator may expand a concern card to read the full message, type a reply in a provided text area, and send the reply to the student's email with a single click. The reply is sent as a background task so the interface remains responsive. Concerns may also be marked as resolved without a reply, or deleted entirely.

Feature 28  —  Chat Logs Viewer with Smart Export

The Chat Logs page gives the administrator full visibility into every question asked through the chat interface, showing the most recent 200 entries. The three summary cards at the top — Total Questions, Answered, and Unanswered — function as clickable filter buttons that filter the list instantly. The administrator can also search by keyword to find specific questions. Each log entry can be expanded to view the AI's full response. The Export button is context-aware: when the "Total Questions" filter is active it exports all logs, when "Answered" is active it exports only answered logs, and when "Unanswered" is active it exports only unanswered logs. The exported CSV file includes the question, answer, status, and timestamp of each entry.

Feature 29  —  Office Processes Management with Auto-Sync

The Office Processes page allows the administrator to add, edit, and delete the office process entries displayed on the landing page. Each entry includes an office name, tagline, icon, color theme, and detailed process sections stored as structured JSON data. Every time an office process is created, updated, or deleted, the system automatically re-synchronizes all office process content into the ChromaDB vector database, ensuring the AI can immediately answer student questions about office procedures based on the most current information without requiring a manual document re-upload. This synchronization also runs once automatically when the backend server starts.

Feature 30  —  Subscriber Management

The Subscribers page lists all students who have subscribed to the announcement email system, showing each subscriber's email address and the date they subscribed. The administrator can remove individual subscribers by clicking the trash icon next to their entry. This allows the administrator to maintain a clean and accurate subscriber list.

Feature 31  —  QR Code Generator with Auto-Detection

The QR Code page generates a scannable QR code that students can use to open the KCCSmartInfoX chatbot on their phone. The system automatically detects the local network IP address of the server and the current browser port from the operating system, so the generated QR code always points to the correct network address without requiring manual configuration. The administrator can download the QR code as a PNG image for printing or display in classrooms, or copy the chatbot URL to the clipboard. Quick preset buttons allow switching between the local network URL for in-campus use and the localhost URL for development testing.

Feature 32  —  Password Change with Verification

The Settings page allows the administrator to update their account password. The password change form requires the administrator to enter their current password for verification, ensuring that only someone who knows the existing password can change it. The new password must be at least six characters long. A real-time visual indicator below the confirm password field shows whether the new password and confirmation match. Passwords are hashed using bcrypt before being stored in the database.

Feature 33  —  Admin Audit Log

The Settings page includes an Admin Audit Log section that displays the last fifty administrator session events in chronological order. Each event is color-coded by type: Login Success events are shown in green, Login Failed events in red, Logged Out events in gray, and Session Timeout events in orange. Each log entry shows the event type, the administrator's username, the device type derived from the user agent string (such as Windows PC, iPhone, or Android), the IP address of the request, and the exact date and time. This audit log provides complete visibility into all admin account activity and helps detect unauthorized access attempts.

Part 3  ·  AI and System Intelligence Features
Feature 34  —  Retrieval-Augmented Generation (RAG) Pipeline

The RAG Pipeline is the core AI mechanism of KCCSmartInfoX. Unlike standard chatbots that generate answers purely from their training data — which may be outdated or hallucinated — the RAG approach first retrieves relevant factual content from official KCC documents before generating a response. This grounding in verified source documents ensures that the AI cannot fabricate school information such as tuition fees, deadlines, or office names. The pipeline consists of thirteen sequential steps: thanks detection, greeting detection, spell correction, LLM query rewriting, semantic embedding, ChromaDB vector search, distance threshold evaluation, context trimming, LLM prompt assembly, Groq API answer generation, bullet formatting cleanup, answered status evaluation, and database logging.

Feature 35  —  Semantic Embedding and Vector Search

When a student submits a question, it is converted into a 384-dimensional numerical vector using the all-MiniLM-L6-v2 sentence-transformers model running locally on the server. This vector represents the semantic meaning of the question rather than just its keywords. The question vector is then compared against all document chunk vectors stored in the ChromaDB vector database using cosine similarity — a mathematical measure of directional similarity between vectors. The three most similar chunks are retrieved. If the closest match has a distance score greater than 1.3, the system determines that no relevant information exists in the knowledge base and marks the question as unanswered.

Feature 36  —  Multilingual Query Understanding

The system is designed to understand student questions written in English, Filipino, Taglish, Bisaya, or any combination thereof. After spell correction, the question is sent to the Groq language model with a specialized rewriting prompt that instructs it to interpret the student's intent and convert the input into a clean, precise English question for vector search. For example, the input "pano kumuha ng tor sa rejistrar" is rewritten as "How do I get a Transcript of Records from the registrar?" — a form that produces significantly more accurate search results against the document embeddings.

Feature 37  —  Conversation History Context

The system maintains context across multiple messages within the same session. The last four messages of the student's conversation history are included in every prompt sent to the Groq language model, allowing the AI to understand follow-up questions that reference previous answers. For example, if a student asks "How do I enroll?" and then asks "What documents do I need?", the AI understands that the second question refers to enrollment documents because the conversation history is included. This context window of four messages balances coherence with token efficiency.

Feature 38  —  Groq API Fallback Error Handling

All calls to the Groq API — both for query rewriting and for answer generation — are wrapped in error handling blocks. If the Groq API is temporarily unavailable, returns a timeout, or encounters any error, the system catches the exception and returns a user-friendly message informing the student that the AI service is temporarily unavailable and asking them to try again in a moment. The retry button displayed alongside the error message allows the student to resubmit their question without retyping. This ensures the system degrades gracefully without exposing raw server errors to the student.

Part 4  ·  Security and Technical Features
Feature 39  —  JWT Authentication with 401 Interceptor

All administrator API endpoints are protected using JSON Web Token (JWT) authentication implemented with the python-jose library. The admin frontend automatically includes the JWT token in the Authorization header of every API request. A global Axios response interceptor monitors all API responses: if any request returns an HTTP 401 Unauthorized status — indicating an expired or invalid token — the interceptor immediately clears the stored token from local storage and redirects the administrator to the login page. This ensures that no sensitive admin data is accessible after a session expires.

Feature 40  —  CORS Security Policy

The backend server enforces a strict Cross-Origin Resource Sharing (CORS) policy that limits which origins are permitted to make requests to the API. Only the frontend development server origins — localhost:5173 and localhost:4173, along with their 127.0.0.1 equivalents — are allowed. This prevents unauthorized web applications from making API calls to the backend from different origins, protecting student data and admin functionality from cross-origin attacks.

Feature 41  —  File Upload Size Limit

The knowledge base document upload endpoints enforce a maximum file size of 20 megabytes per file, applied to both single and bulk uploads. Files that exceed this limit are rejected before being saved to the server's filesystem, and an HTTP 413 error response is returned with an appropriate message. This prevents large files from consuming excessive server storage and ensures the document processing pipeline operates within acceptable performance bounds.

Feature 42  —  Chat Endpoint Rate Limiting

The public chat endpoint is rate-limited to twenty requests per minute per IP address using the slowapi library. This prevents automated bots or scripts from flooding the system with queries and consuming Groq API credits. Users who exceed the rate limit receive an HTTP 429 Too Many Requests response. The rate limit is applied per IP address, so it does not affect other legitimate users on the network.

Feature 43  —  404 Not Found Page

Any URL that does not match a valid route in the system is handled by a custom 404 Not Found page rather than the browser's default error display. The page shows the KCC logo, a clear "Page not found" message, and two navigation buttons: one to return to the Landing Page and one to open the Chatbot. This ensures that students who navigate to an incorrect URL are guided back to the correct section of the system without confusion.

Feature 44  —  ChromaDB Singleton Client

The ChromaDB vector database client is instantiated as a shared singleton throughout the entire backend application. This design prevents database lock conflicts that would occur if multiple parts of the application — such as the document upload endpoint and the chat endpoint — attempted to open the database simultaneously. The singleton ensures that all read and write operations share the same database connection, eliminating the SQLite lock errors that caused intermittent failures in earlier implementations.
