Platform: Railway  ·  Strategy: Single-Service Deployment  ·  Timeline: This Week

★
Chosen Platform: Railway

Why Railway and not others? The frontend uses relative API paths (/api/...) with no base URL — meaning the frontend and backend must be served from the same domain. Railway lets us do this with one service. Render's free tier sleeps after 15 minutes of inactivity (causes a 30–60 second delay during demo). Vercel and Netlify are frontend-only. Railway wins: always-on, single URL, deploys from GitHub, free starter credit.

Code Changes Required (4 Files Only)

Minimal changes — the system already works locally; these changes adapt it for cloud hosting.

# File What Changes Why
1 backend/requirements.txt Remove sendgrid line sendgrid is listed but never used (Gmail SMTP is used instead). Can cause Railway install errors.
2 backend/main.py Add 6 lines at the bottom — FastAPI serves the built React app as static files + SPA catch-all route Since all axios calls use relative /api/... paths, the frontend must be served from the same origin as the backend. This is the cleanest solution with zero changes to the frontend.
3 railway.toml NEW FILE Build command: install frontend → build React → install backend. Start command: uvicorn on Railway's dynamic port. Railway needs to know how to build a project with both a Python backend and a Node/React frontend in the same repo.
4 .gitignore Ensure kccsmartinfox.db and knowledge_base/vectorstore/ are not ignored If the database and AI vectorstore are excluded from git, Railway will deploy with no data — AI cannot answer anything and the admin panel will be empty.

Code added to backend/main.py (bottom of file):

import pathlib
from fastapi.responses import FileResponse

_dist = pathlib.Path(__file__).parent.parent / "frontend" / "dist"
if _dist.exists():
    app.mount("/assets", StaticFiles(directory=str(_dist / "assets")), name="spa-assets")

    @app.get("/{full_path:path}", include_in_schema=False)
    async def serve_spa(full_path: str):
        return FileResponse(str(_dist / "index.html"))

Contents of new railway.toml:

[build]
builder = "nixpacks"
buildCommand = "cd frontend && npm install && npm run build && cd ../backend && pip install -r requirements.txt"

[deploy]
startCommand = "cd backend && uvicorn main:app --host 0.0.0.0 --port $PORT"
healthcheckPath = "/api/stats"
healthcheckTimeout = 60
restartPolicyType = "on_failure"

Step-by-Step Deployment Process

One-time setup. Do this the day before the defense — not on demo day.

1
Apply the 4 code changes
Make the changes to requirements.txt, main.py, create railway.toml, and fix .gitignore. Commit and push everything to GitHub including the database and vectorstore.
2
Create a Railway account
Go to railway.app → Sign up with GitHub. This links Railway directly to your repository — no manual file uploads needed.
3
Create a new project from GitHub
Click New Project → Deploy from GitHub repo → select KaisuS12/KCCSmartInfoX. Railway reads railway.toml automatically and starts building.
4
Set environment variables
In Railway → your project → Variables tab → add the 4 variables from Section 2 above. Railway will trigger a new build automatically.
5
Generate a public domain
Go to Settings → Networking → Generate Domain. Railway gives you a .up.railway.app URL. This is your demo URL. Copy it and save it.
6
Wait for build to complete (~4–6 minutes)
The first build takes longer because sentence-transformers (~500MB) is downloaded. Watch the Railway build logs. When it shows a green checkmark, the system is live.
7
Test everything before the defense
Open the URL → test chat → test admin login → ask a question the AI knows → submit a test concern. Confirm emails work. Share the URL with your group.

Pre-Demo Checklist — Fix These Before Defense Day

These issues will cause the demo to fail or look bad in front of panelists if not addressed.

# Issue Risk Level How to Fix
1 ChromaDB vectorstore may be excluded from git CRITICAL Check .gitignore — if vectorstore/ is listed, remove it, then commit and push the folder. Without it, the AI cannot answer any question.
2 SQLite database may be excluded from git CRITICAL Check .gitignore — if *.db is listed, remove it. Without the DB, the admin panel is empty and has no data to show.
3 Groq API quota may be used up HIGH Log in to console.groq.com and check your usage. The free tier has rate limits. If near the limit, create a new free Groq account and update the API key.
4 Do a full test deploy the night before HIGH Never deploy for the first time on demo day. Deploy the night before, test all features, then leave it running. The first build takes 4–6 minutes.
5 Admin password confirmation MEDIUM Log in to the live URL with admin / admin123 before the defense. Confirm it works on the deployed version, not just locally.
6 Do not redeploy during or before the defense MEDIUM Railway resets the filesystem on redeploy. Any files uploaded during the demo (documents, images) will be lost. Upload demo content before the final deploy.
7 Open the URL 5 minutes before presenting LOW Keep a browser tab open to prevent any idle timeout. Pre-load the chat page so the model is warm and the first response is fast.
8 Gmail app password still valid LOW Test the email feature on the live URL by submitting a concern. If email fails, regenerate the Gmail app password and update the Railway variable.
