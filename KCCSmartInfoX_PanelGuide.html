Figure 1
Conceptual Framework — IPO Model
https://www.figma.com/board/wJaK19VPcmv5sXY8bNHZ3o
What this diagram shows

The overall Input–Process–Output model of the entire KCCSmartInfoX system. It shows what data goes in, what the system does with it, and what comes out.

🎤 How to explain to the panel

"This is our Conceptual Framework using the IPO model — Input, Process, and Output. On the INPUT side, we have the student's question, their chosen language which is English or Filipino, voice input, and feedback. On the admin side, the inputs are the uploaded documents, announcements, and office processes. These inputs go into our PROCESS layer where the system performs authentication, spell correction, AI query rewriting, semantic embedding, vector search, and LLM answer generation. The OUTPUT is the AI-generated answer with source citations for the student, and analytics reports, chat logs, and audit logs for the administrator. This framework guided the entire design of our system."

✅ Key points to remember
  • IPO means Input → Process → Output
  • Two types of users feed inputs: Student and Admin
  • The Process is where the AI happens (RAG pipeline)
  • Outputs go back to both users in different forms
❓ Possible panelist questions
Q: Why did you use the IPO model as your framework?
A: "The IPO model is the most appropriate framework for information systems because our system is fundamentally about receiving data, processing it through AI, and returning useful output. It clearly shows the flow of information from both the student and administrator perspectives."
Q: What is the most important part of your process?
A: "The RAG Pipeline — Retrieval-Augmented Generation. It is what makes our AI answer accurately based only on official KCC documents, rather than generating unreliable information."
Figure 2
System Architecture Diagram
https://www.figma.com/board/zQ3maIg3vfvfAXopGnlpbi
What this diagram shows

The three-tier architecture of the system — how the browser/phone, the backend server, and the databases are connected and communicate with each other.

🎤 How to explain to the panel

"This is our System Architecture Diagram. Our system follows a three-tier architecture. The first tier is the Client Layer — this is what the student or admin sees in their browser or phone, built using React 18 and Vite. The second tier is the Backend Layer — this is the FastAPI server running in Python that handles all requests, manages security through JWT authentication, and coordinates the AI pipeline. The third tier is the Data Layer — this includes the SQLite relational database for structured data like chat logs and announcements, the ChromaDB vector database for storing document embeddings, and the external Groq API for AI answer generation. The client communicates with the backend through HTTP REST API calls, and the backend connects to the data layer to retrieve and store information."

✅ Key points to remember
  • Three tiers: Client → Backend → Data
  • Frontend: React 18 + Vite + TailwindCSS
  • Backend: FastAPI + Python + Uvicorn
  • Two databases: SQLite (structured data) + ChromaDB (AI embeddings)
  • One external AI service: Groq API
❓ Possible panelist questions
Q: Why did you use two databases?
A: "We needed two different types of storage. SQLite is a relational database — good for structured records like user accounts, chat logs, and announcements. ChromaDB is a vector database — it stores mathematical representations of text called embeddings, which are needed for the AI to find relevant documents by meaning, not just keywords."
Q: Why FastAPI and not Django or Node.js?
A: "FastAPI is the best choice for AI applications because it is asynchronous by design, which means it can handle multiple users at the same time. It also has automatic API documentation and is written in Python — the same language as our AI libraries like sentence-transformers and ChromaDB."

❓ Possible panelist questions
Q: Why is Groq API an external entity?
A: "Because Groq is a third-party cloud service that our system depends on but does not control. Our system sends a prompt to Groq's servers and receives a generated text response. It is external because it exists outside our system boundary."
Q: What if Groq API is unavailable?
A: "Our system has error handling for this. If Groq is unreachable, the student sees a friendly message asking them to try again in a moment instead of a raw error. The system does not crash."
Figure 5
Data Flow Diagram — Level 1
https://www.figma.com/board/Rn8DQWBoNNPqcDguBcGhNr
What this diagram shows

The system broken down into its six main internal processes, showing how data flows between each process, the databases, and the external entities.

🎤 How to explain to the panel

"This is our DFD Level 1. Here we open up the system and show the six main internal processes. Process 1 is Authentication — the administrator logs in, the system verifies credentials against the SQLite database and returns a JWT token. Process 2 is Chat Processing, which is our RAG Pipeline — the student sends a question, the system searches ChromaDB for relevant documents, sends a prompt to the Groq API, and returns the AI answer. All questions and answers are stored in SQLite. Process 3 is Knowledge Base Management — the admin uploads documents, the system extracts text, creates embeddings, and stores them in both SQLite and ChromaDB. Process 4 is Announcement Management — the admin creates announcements stored in SQLite, and the system sends emails to subscribers through Gmail SMTP. Process 5 is Analytics and Reporting — the admin requests reports, the system reads from SQLite and returns statistics and CSV exports. Process 6 is Concern Management — students submit concerns stored in SQLite, admins reply, and the reply is sent to the student via Gmail SMTP."

✅ Key points to remember
  • 6 main processes: Auth, Chat/RAG, Knowledge Base, Announcements, Analytics, Concerns
  • Two data stores: SQLite (structured) and ChromaDB (vectors)
  • Process 2 (Chat/RAG) is the core and most complex process
  • Gmail SMTP is used by both Announcements (Process 4) and Concerns (Process 6)
❓ Possible panelist questions
Q: Why does Process 3 write to both databases?
A: "SQLite stores the document metadata — the filename, file path, and upload date — so the admin can see what documents are in the system. ChromaDB stores the actual text embeddings of the document chunks, which are needed for the AI to search by meaning. Both are needed for the knowledge base to work completely."
Figure 6
RAG Pipeline Flowchart — AI Query Processing
https://www.figma.com/board/U2vTNfVxLyPvXX4dMLwUs7
What this diagram shows

The complete step-by-step process of how the AI handles a student's question — from the moment they press send to the moment the answer appears on screen. This is the heart of the system.

🎤 How to explain to the panel

"This flowchart shows how our AI processes every student question — this is called our RAG Pipeline, or Retrieval-Augmented Generation Pipeline. When a student sends a message, the system first checks if it is a thank-you — like salamat or thanks — and if so, it replies instantly without calling any AI. Next it checks if it is a greeting like hello or kumusta, and if so, it sends a simple conversational response. If it is a real question, the system first corrects any spelling mistakes using a local spell checker that knows Filipino and KCC-specific words. Then it sends the corrected question to the Groq AI to rewrite it into clean English — this helps the AI understand Taglish and abbreviations. The clean question is then converted into a mathematical vector using our local embedding model. This vector is compared against all the document vectors stored in ChromaDB. If the closest match is too far away in meaning, the question is marked as unanswered. If relevant documents are found, they are used as context and sent together with the question to the Groq AI which generates the final answer. The answer is cleaned, checked, logged to the database, and returned to the student."

✅ Key points to remember
  • 13 steps total from question to answer
  • Thanks and greetings are handled without any document search
  • Spell fix is local — no internet needed for this step
  • Query rewriting converts Taglish/Bisaya to clean English
  • Distance threshold 1.3 — if score is higher, question is unanswered
  • Groq API is called only twice: for query rewriting and for answer generation
  • All questions logged to database regardless of answered status
❓ Possible panelist questions
Q: What is RAG and why did you use it?
A: "RAG stands for Retrieval-Augmented Generation. Instead of letting the AI generate answers from its training data — which can be outdated or hallucinated — RAG first retrieves relevant information from our trusted KCC documents and uses that as context for the AI to answer from. This ensures our AI only answers based on official KCC information and cannot make up fees, dates, or policies that don't exist."
Q: How does the system understand Filipino questions?
A: "Through two steps. First, the spell checker is trained to preserve Filipino and Bisaya words so they are never incorrectly changed. Second, the query rewriting step uses the Groq AI to understand the intent of a Taglish or Bisaya message and convert it into a clean English question. For example, 'pano kumuha ng tor sa rejistrar' becomes 'How do I get a Transcript of Records from the registrar?' — which then searches the documents much more accurately."
Q: What happens when the AI cannot answer?
A: "Two things happen. First, the student is shown the AI's honest response saying it does not have that information yet. Second, the question is logged to the database as unanswered and visible to the admin in the Analytics page and Dashboard. The admin can then add the missing information to the knowledge base. The system also shows a contact card so the student can submit a formal concern."
Figure 7
Entity Relationship Diagram (ERD)
https://www.figma.com/board/wN2r1Rgjj1vwVef2T4Jc64
What this diagram shows

All 10 database tables used by KCCSmartInfoX, the columns inside each table, and the relationships between related tables.

🎤 How to explain to the panel

"This is our Entity Relationship Diagram, or ERD. It shows the structure of our SQLite database. We have 10 tables in total. The admin_users table stores the administrator's login credentials with bcrypt-hashed passwords. The chat_logs table records every question and answer including whether it was answered or not. The feedback table stores the thumbs-up and thumbs-down ratings from students, linked to a specific chat log through a foreign key. The subscribers table stores student email addresses for announcement notifications. The announcements table holds all announcement content with scheduling and expiry dates. The knowledge_docs table keeps track of all uploaded documents and manual text entries. The office_processes table stores the office process cards shown on the landing page and synced to the AI. The concerns table stores student concerns and admin replies. The admin_login_logs table records every login, logout, and session timeout event for the audit log. And finally the admin_ai_logs table records Admin AI Assistant actions. The main relationships are: feedback is linked to chat_logs, and concerns may be related to a chat log entry."

✅ Key points to remember
  • 10 tables total in SQLite
  • Passwords are never stored in plain text — always bcrypt hashed
  • feedback references chat_logs via foreign key (chat_log_id)
  • concerns has a status column: "pending" or "resolved"
  • admin_login_logs tracks: success, failed, logout, timeout
  • office_processes sections stored as JSON inside a single column
❓ Possible panelist questions
Q: Why SQLite and not MySQL or PostgreSQL?
A: "SQLite is ideal for our system's current scope because it is file-based, requires no separate database server, and is easy to deploy on any machine. For a school-level system with hundreds of users, SQLite is more than capable. If the system needs to scale to thousands of concurrent users in the future, migration to PostgreSQL is straightforward because we use SQLAlchemy as our ORM, which supports multiple database backends."
Q: How is the password security handled?
A: "Administrator passwords are hashed using bcrypt before being stored in the database. Bcrypt is an industry-standard one-way hashing algorithm — meaning the original password can never be recovered from the hash. During login, the submitted password is compared against the hash using passlib's verify function."
