Section 2  ·  Backend Processes — One by One
2.1   Process 1: Admin Authentication

File: backend/routes/admin.py  ·  Endpoint: POST /api/admin/login

When the administrator submits their credentials, the system first checks the rate limiter — only 5 login attempts per minute per IP are allowed. If under the limit, the system fetches the AdminUser record from SQLite and verifies the submitted password against the stored bcrypt hash using passlib. On success, a JWT token is generated with a 24-hour expiration and returned to the frontend. Both successful and failed login attempts are recorded in the admin_login_logs table with the IP address, user agent, and timestamp.

# Actual flow in admin.py: @limiter.limit("5/minute") # Rate limiting → verify_password(plain, user.password_hash) # bcrypt check → create_access_token({"sub": username}) # JWT 24h → db.add(AdminLoginLog(action="success")) # Audit log
2.2   Process 2: RAG Chat Pipeline

File: backend/rag/pipeline.py  ·  Endpoint: POST /api/chat

Step 1 — Thanks Detection

The is_thanks() function checks the student's message against a regex pattern covering English and Filipino gratitude expressions (thanks, salamat, maraming salamat, daghan salamat, ty, thx, etc.). If matched, one of five randomized warm reply messages is returned immediately — no API call needed.

Step 2 — Greeting Detection

The is_greeting() function checks against casual messages (hi, hello, kumusta, good morning, bye, etc.). If matched, the message is sent directly to Groq with a conversational prompt (temperature=0.7, max_tokens=150). A try/except fallback returns a default message if Groq is unavailable.

Step 3 — Spell Correction

The quick_spell_fix() function runs the student's question through pyspellchecker word by word. A custom preserve list prevents over-correction of Filipino/Bisaya words and KCC-specific terms (kcc, tor, bsit, salamat, po, ate, kuya, etc.).

Step 4 — LLM Query Rewriting

The rewrite_query() function sends the spell-corrected question to Groq with a specialized rewriting prompt at temperature=0.0. The model converts Taglish, Bisaya, typos, and abbreviations into clean English. For example: "pano kumuha ng tor sa rejistrar" becomes "How do I get a transcript of records from the registrar?"

Step 5 — Semantic Embedding

The rewritten clean question is encoded using the all-MiniLM-L6-v2 sentence-transformers model running locally on the server. The model outputs a 384-dimensional float vector representing the semantic meaning of the question.

Step 6 — ChromaDB Vector Search

The question embedding is queried against all stored document chunk embeddings in ChromaDB using cosine similarity. The top 3 closest chunks are retrieved along with their distance scores and source metadata. If the closest distance exceeds 1.3, no relevant context exists and the question is flagged as unanswered.

Step 7 — Context Building

Each retrieved chunk is trimmed to a maximum of 150 words to stay within Groq's token limits. The chunks are joined with separators and unique source filenames are extracted from the metadata for citation display.

Step 8 — Groq LLM Answer Generation

A prompt is assembled from the system instructions, the last 4 messages of conversation history, the RAG context, and the student's question. This is sent to Groq's llama-3.1-8b-instant model at temperature=0.2 with a max of 400 output tokens. A try/except fallback returns a user-friendly error message if the Groq API fails.

Step 9 — Post-Processing and Logging

The answer is passed through clean_bullets() which replaces Unicode bullet characters (•·●▪◆) with proper markdown "- " format. The system then checks whether the phrase "I don't have information about that yet" appears in the answer to set the is_answered flag. The question, answer, and flag are saved to the chat_logs table in SQLite.

2.3   Process 3: Document Ingestion

File: backend/rag/ingestion.py  ·  Endpoints: POST /api/admin/knowledge/upload, /bulk-upload, /add-text

When the administrator uploads a document, the system first enforces a 20 MB file size limit. For PDF files, pdfplumber extracts text page by page. For DOCX files, python-docx reads each paragraph. For TXT files, the content is read directly. The extracted text is then split into overlapping chunks of approximately 400 words with a 50-word overlap to preserve context across boundaries. Each chunk is embedded using the all-MiniLM-L6-v2 model and stored in ChromaDB with the source filename as metadata. The document metadata is also saved to the knowledge_docs table in SQLite.

2.4   Process 4: Announcement Management

File: backend/routes/announcements.py  ·  Endpoints: POST/PUT/DELETE /api/admin/announcements

When an announcement is created, the administrator can set an optional publish_at datetime for scheduled posting and an expires_at datetime for automatic expiry. If publish_at is null, the announcement is published immediately. A background asyncio task runs every 60 seconds to check for scheduled announcements whose publish time has passed and sends the email to all subscribers. Email is sent using Python's built-in smtplib library via Gmail SMTP SSL on port 465. The email is dispatched as a FastAPI BackgroundTask so the API responds immediately without waiting for the SMTP operation to complete. The email_sent flag is set to True after successful delivery.

2.5   Process 5: Office Processes Auto-Sync

File: backend/routes/office_processes.py

Every time an office process is created, updated, or deleted through the admin panel, the system automatically re-ingests all office processes into ChromaDB. This ensures the AI can always answer student questions about office procedures using the most current information without requiring a manual document upload. On application startup, this sync also runs once to ensure the vector store is consistent with the database from the very first request.

2.6   Process 6: Analytics and Reporting

File: backend/routes/admin.py  ·  Endpoint: GET /api/admin/analytics

The analytics endpoint queries the SQLite database to compute: total question count, answered count, unanswered count, answer rate percentage, total subscriber count, total document count, thumbs-up count, and a list of recent unanswered questions. The FAQ Auto-Generator feature sends the top unanswered questions to Groq and receives suggested knowledge base entries that the administrator can add with one click. The chat log export endpoint accepts a filter parameter (all/answered/unanswered) and streams a CSV file with ID, question, answer, status, and timestamp columns.

2.7   Process 7: Concern Management

File: backend/routes/admin.py  ·  Endpoints: PUT /api/admin/concerns/{id}/reply, /resolve

When a student submits a concern through the chat interface, it is saved to the concerns table with a status of "pending." An automated email is sent to the administrator notifying them of the new concern. When the administrator replies, the reply text is saved to the admin_reply column, the status is updated to "resolved," the replied_at timestamp is recorded, and the reply is sent to the student's email address using Gmail SMTP as a background task.

2.8   Process 8: Admin Session Management

File: frontend/src/components/shared/AdminLayout.jsx

JWT tokens expire after 24 hours. A global Axios response interceptor detects any 401 Unauthorized response and immediately redirects to the login page. Additionally, the admin panel tracks five user activity events — mousemove, keydown, mousedown, touchstart, and scroll. If no activity is detected for 30 consecutive minutes, the session is terminated, a "timeout" event is logged to admin_login_logs via the POST /api/admin/logout-log endpoint, and the administrator is redirected to the login page.

Section 4  ·  UI Pages Review
4.1   User-Facing Pages
Landing Page

The landing page is the first screen students see. It contains a navigation bar with the KCC logo, a hero section with a call-to-action button to open the chatbot, an announcement carousel showing active announcements from the database, a grid of office process cards, and a footer. All announcement and office process data is fetched dynamically from the backend API.

Chat Page

The chat page is the core student interface. In its empty state it displays the KCC logo (with a floating animation), four suggested starter questions, and a text input with a microphone button. The header contains the language toggle (EN/FIL), an Announcements button, and navigation back to the landing page. A character counter below the input shows the remaining characters up to the 500-character limit. On mobile devices the page adapts to a full-screen chat layout.

Chat Page — AI Answer State

After a student submits a question, a typing indicator appears while the AI processes the response. The AI's answer is rendered using ReactMarkdown, which formats bullet points, bold text, and numbered lists. Below the answer, source document filenames are displayed for citation. Thumbs-up and thumbs-down feedback buttons allow the student to rate the response. If the AI could not answer, a contact card appears with a Submit Concern button. If an error occurs, a retry button allows the student to resend the question automatically.

4.2   Admin Panel Pages
Admin Login

A centered card with the KCC logo, username and password fields (with show/hide toggle), and a Login button. Rate-limited to 5 attempts per minute. All attempts logged in the audit log.

Dashboard

Displays 8 stat cards (Total Questions, Answered, Unanswered, Answer Rate, Subscribers, Documents, Helpful, Pending Concerns), a Backend Online/Offline status pill, quick-action buttons, and a list of recent unanswered questions. Auto-refreshes every 10 seconds.

Knowledge Base

Upload PDF/DOCX/TXT files individually or in bulk (up to 20 MB each). Add text entries directly. Inline edit existing text entries (old content removed from ChromaDB, new content indexed). Delete documents (removed from DB and ChromaDB simultaneously).

Announcements

Create announcements with title, content, optional image, optional publish date/time, and optional expiry date. Edit existing announcements via pencil icon. Delete announcements. All active announcements visible to students on the landing page and chat panel.

Analytics

Daily questions bar chart (last 7 days), top 10 most-asked questions, all unanswered questions list, negatively-rated answers. FAQ Auto-Generator button analyzes unanswered questions using Groq and suggests knowledge base content. Export CSV filtered by all/answered/unanswered.

Chat Logs

Three clickable filter stat cards (Total Questions / Answered / Unanswered). Search by keyword. Expand each log to view full AI answer. Export button dynamically labeled based on active filter (Export All / Export Answered / Export Unanswered).

Student Concerns

Three clickable filter stat cards (All Concerns / Pending / Resolved). Expand concern cards to view full message, type a reply, send reply via email, or mark as resolved without reply. Delete concerns. Reply is sent to student email as a background task.

Settings

Two-column layout: Change Password form (left) with current password verification and match indicator; Admin Audit Log (right) showing last 50 session events with color-coded event types (Login Success / Login Failed / Logged Out / Session Timeout), device type, IP address, and timestamp.

QR Code

Auto-detects the local network IP and current browser port using window.location. Generates a scannable QR code pointing to the chat URL. Download as PNG or copy URL to clipboard. Quick preset buttons for network and localhost access.
